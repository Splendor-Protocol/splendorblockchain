## üìù Changelog Format

Our changelog follows the [Keep a Changelog](https://keepachangelog.com/en/1.0.0/) format:

### Version Structure
```
## [Version] - YYYY-MM-DD

### Added
- New features

### Changed
- Changes to existing functionality

### Deprecated
- Soon-to-be removed features

### Removed
- Removed features

### Fixed
- Bug fixes

### Security
- Security updates
```

## üîÑ Version Numbering

We use [Semantic Versioning](https://semver.org/):

- **MAJOR** version (X.0.0) - Incompatible API changes
- **MINOR** version (0.X.0) - Backwards-compatible functionality additions
- **PATCH** version (0.0.X) - Backwards-compatible bug fixes

## ü§ù Contributing to Changelog

When making changes to the project:

1. **Document your changes** in [CHANGELOG.md](CHANGELOG.md)
2. **Use the correct category** (Added, Changed, Fixed, etc.)
3. **Be descriptive** but concise
4. **Include issue/PR references** when applicable
5. **Follow the format** consistently

Example:
```markdown
### Added
- X402 payment protocol support (#123)
- New RPC endpoint for balance queries (#124)

### Fixed
- Transaction broadcasting issue (#125)
- Memory leak in consensus module (#126)
```
